技術的なポイント

IDEA AgentはClaude CodeというAIツール上に、ルール（.mdファイル）やSkillを定義して動かしています。
この「.mdファイルで作り込んだAIの振る舞い」をそのまま使えるかどうかが、デプロイ方法を選ぶ最大の判断基準です。

要するに、「今ローカルPCで動いているものを、どこで24時間動かすか」という問題です。

Details

各選択肢の詳細

推奨（本番運用）

A. 社内PC（常時稼働機）+ Claude Code Remote

社内ネットワークにPCを1台設置（Windows / Mac どちらでもOK）。
今ローカルPCで動かしているIDEA Agentの環境をそのまま移設するイメージ。
外部（tldv）からの信号を受け取り、AIが自動で処理を実行。

メリット

  • 今作り込んだルール・Skillがそのまま動く
  • .mdファイルを編集するだけで機能追加・修正が簡単
  • エンジニア以外でもルール追記が可能
  • データが社内に留まる（セキュリティ面で有利）
  • 社内に余っているPCがあれば初期費用ゼロ

デメリット・リスク

  • ハードウェアの管理が必要（故障・停電対応）
  • 常時起動のため電源・ネットワークの確保が必要
  • ネットワーク設定が必要（社内NW担当と連携）
将来候補

B. Cloud Run + Agent SDK

Google Cloud上にプログラムを配置。
MTGの信号を受け取ったときだけ起動するサーバーレス方式。
ハードウェア不要、Google側が管理。

メリット

  • ハードウェア管理が不要
  • 使った分だけ課金（アイドル時はほぼ0円）
  • Google Cloudの高い安定性
  • 将来的にスケールしやすい

デメリット・リスク

  • 今の.mdルールをPythonコードに書き直す必要がある
  • 開発工数が大きい（フルリライト）
  • 変更のたびにエンジニアの作業が必要
PoC（まず試す用）

C. GAS + Claude API

Google Apps Script（GAS）でWebhookを受け取り、
Claude APIを呼んで処理する最小構成。
まず動くかどうかを確認するためのPoC（実証実験）向け。

メリット

  • 初期費用ゼロ、すぐ始められる
  • 開発が簡単（1〜2日で構築可能）
  • Google Workspaceの既存環境で動く
  • ChatWork通知・GitHub保存は十分対応可能

デメリット・リスク

  • 1回の実行が6分まで（長い文字起こしは分割が必要）
  • 複雑なエージェント処理には不向き
  • スケールには限界がある
代替案

D. クラウドVM（VPS）

AWS/GCPで仮想マシンを月額契約し、24時間起動しっぱなしにする。
その中にClaude Codeをインストールし、.mdルール・Skillを配置。
社内PCと同じ動かし方だが、物理的な機器が不要。

メリット

  • ハードウェア故障リスクなし
  • リモートアクセスが容易
  • 今の.mdルールがそのまま使える

デメリット・リスク

  • 月額費用が高め（月5,000〜15,000円）
  • 常時稼働のため無駄が出やすい
  • macOS VMは提供が限られる

Roadmap

考えている進め方

いきなり本番環境を構築するのではなく、段階的に進めます。
最終的には社内のMacBook Proを常時稼働機として、フルスペックのエージェントを動かします。

STEP 1: GASでPoC（1〜2日）
最小構成で「自動処理が動く」ことを実証
tldv Webhook → GAS → GitHubに文字起こし保存 + ChatWork通知
費用ゼロ、開発1〜2日で構築可能。
検証すること: tldv Webhookが正しく飛ぶか、文字起こし取得→保存→通知の一連の流れ
STEP 2: GASにAI要約を追加（1〜2週間）
Claude APIで要約・タスク抽出を自動化

ChatWorkに「MTG要約 + タスク一覧」を自動投稿。
ここでの限界: GASは6分実行制限があり、エージェントの複雑な振る舞い（Skill連携・Drive参照・複数ステップ処理）は実現できない
STEP 3: 実機（MacBook Pro）でフル稼働
IDEA Agentのフルスペックを常時稼働
社内のMacBook Proにリポジトリ + Claude Code環境を構築。

Webhook受信 → AIエージェントが自動処理 → 企画書ドラフト・調査レポートまで自動生成。
GASでは不可能だった高度なエージェント処理が実現可能に。

GASで小さく動かして検証 → MacBook Proでフル展開

まずGASで費用ゼロ・最短で「MTGが終わったら自動で通知が来る」を実証します。
チームの反応を見ながら、最終的にはMacBook Proでエージェントのフル機能を稼働させます。

技術的なポイント

Q. tldvのWebhookをどうやってMacBook Proで受け取る？
MacBook Pro上にローカルAPIサーバー（Flask/Expressなど）を立ち上げます。
ただし、そのままではインターネットからアクセスできないため、トンネルサービス（Cloudflare Tunnel等）を使って外部から到達可能にします。
※ Cloudflare Tunnelは無料で利用可能。社内PCに固定URLを付与するサービスです。
Q. WebhookからClaude Codeをどう起動する？
ローカルAPIサーバーがWebhookを受け取ったら、Claude Code CLIをコマンドとして実行します。
claude -p "このMTGの文字起こしを処理して"
このとき、.claude/ 配下のルール・Skillが全て読み込まれるので、今ローカルで動かしているのと全く同じ振る舞いをします。
Q. GitHub・Google Drive・ChatWorkとの連携は？
全て既に構築済みの仕組みをそのまま使います。
・GitHub: git commit + push（Claude Codeが自動実行）
・Google Drive: サービスアカウント経由（Python スクリプト、構築済み）
・ChatWork: ChatWork API でメッセージ送信（APIトークンのみ必要）
新しく作る部分はWebhook受信サーバーだけです。
Q. MacBook Proが落ちたらどうなる？
停電・再起動が起きると処理が止まります。対策として：
・macOSの自動ログイン + サービス自動起動を設定（再起動後に自動復旧）
・Cloudflare Tunnelも自動起動設定可能
・万が一止まっても、Webhookの再送やキューイングで取りこぼしを防ぐ仕組みを入れられる
※ 頻繁に落ちるようであれば、その時点でクラウドVMへの移行を検討
Q. なぜGASだけでは不十分なのか？
GASは「要約してChatWorkに通知」程度なら問題なく動きます。
しかし、IDEA Agentが目指すエージェントとしての複雑な処理は実現できません：
・過去のMTG履歴を踏まえた文脈理解（Skill連携）
・Google Driveの成果物を参照した企画書ドラフト作成
・複数ステップの逐次処理（分析→判断→生成→保存→通知）
・6分の実行時間制限（長いMTGの処理が間に合わない場合がある）
これらはClaude Code + .mdルール・Skillの環境でしか実現できません。
